Architecture Series #3: Ports & Adapters Pattern

Decouple your business logic from everything else

Anas Issath
15 min read
·
Mar 2, 2026

I’ve seen this scenario many times:

Developer: “We need to switch from PostgreSQL to MongoDB.”
System: Requires rewriting 50% of the codebase

Developer: “We need to add a GraphQL API.”
System: Business logic is tangled in REST views

Developer: “We need to test without the database.”
System: Impossible — everything depends on Django ORM

The root cause: Business logic married to infrastructure.

The pattern that solves this: Ports & Adapters (Hexagonal Architecture)

Not a Medium member? You can read this article for free via this link: Friend Link

What is Hexagonal Architecture?

A pattern where:
- Business logic is at the center
- Everything else is on the outside
- Interfaces (ports) define boundaries
- Implementations (adapters) plug in

Business logic knows NOTHING about:
- Databases
- Web frameworks
- External APIs
- Message queues
- Anything infrastructure

In 2026, this enables:

  • Test without Django
  • Swap databases without rewriting
  • Add new APIs (REST, GraphQL, gRPC) without touching business logic
  • Change frameworks without breaking domain
  • Deploy to different environments easily

Today, I’ll show you how to implement hexagonal architecture in Django — creating systems where business logic is truly independent.

Real code. Real boundaries. Real freedom.

Let’s decouple everything.

The Problem: Coupled Architecture

Traditional Django Approach

Everything depends on Django ORM:

# views.py
from django.shortcuts import get_object_or_404
from .models import Order, Product
import stripe

def create_order(request):
    # Business logic mixed with infrastructure
    
    # 1. Get product (Django ORM)
    product = get_object_or_404(Product, id=request.POST['product_id'])
    
    # 2. Business validation (tangled with ORM)
    if product.stock < int(request.POST['quantity']):
        return JsonResponse({'error': 'Out of stock'}, status=400)
    
    # 3. Create order (Django ORM)
    order = Order.objects.create(
        user=request.user,
        product=product,
        quantity=request.POST['quantity'],
        total=product.price * int(request.POST['quantity'])
    )
    
    # 4. Update stock (Django ORM)
    product.stock -= order.quantity
    product.save()
    
    # 5. Charge customer (Stripe API)
    stripe.Charge.create(
        amount=int(order.total * 100),
        currency='usd',
        customer=request.user.stripe_id
    )
    
    # 6. Send email (external service)
    send_email(request.user.email, 'Order confirmed')
    
    return JsonResponse({'order_id': order.id})

Problems:

1. Cannot test without Django
   - Business logic needs Django ORM
   - Cannot run tests in plain Python
   - Slow tests (need database)

2. Cannot change database
   - Business logic uses ORM methods
   - Switching to MongoDB = rewrite everything
   - Locked into PostgreSQL

3. Cannot change payment provider
   - Stripe calls in business logic
   - Switching to PayPal = change business code
   - Should be swappable

4. Cannot add new APIs easily
   - Logic mixed in HTTP view
   - GraphQL? Need to duplicate logic
   - gRPC? Duplicate again

5. Business rules scattered
   - Stock validation in view
   - Pricing in model
   - Logic everywhere
   - Hard to find business rules

Direction of dependencies:

The Solution: Hexagonal Architecture

The Hexagon Diagram

Key principles:
1. Business logic in center (no dependencies)
2. Ports define interfaces (abstract)
3. Adapters implement interfaces (concrete)
4. Dependencies point INWARD
5. Business logic knows nothing about outside

Step 1: Define Domain Model

# domain/models/order.py
from dataclasses import dataclass, field
from decimal import Decimal
from enum import Enum
from uuid import UUID, uuid4
from datetime import datetime

class OrderStatus(Enum):
    PENDING = "pending"
    PAID = "paid"
    SHIPPED = "shipped"
    CANCELLED = "cancelled"


@dataclass
class Order:
    """Pure domain model
    
    No Django dependencies
    No database knowledge
    Pure business logic
    """
    id: UUID = field(default_factory=uuid4)
    customer_id: UUID = field(default=None)
    items: list['OrderItem'] = field(default_factory=list)
    status: OrderStatus = field(default=OrderStatus.PENDING)
    created_at: datetime = field(default_factory=datetime.utcnow)
    
    # Domain events
    _events: list = field(default_factory=list, repr=False)
    
    @staticmethod
    def create(customer_id: UUID, items: list['OrderItem']) -> 'Order':
        """Factory method - business rules enforced"""
        if not items:
            raise ValueError("Order must have at least one item")
        
        order = Order(
            customer_id=customer_id,
            items=items
        )
        
        # Domain event
        from domain.events import OrderCreated
        order._events.append(OrderCreated(
            order_id=order.id,
            customer_id=customer_id,
            total=order.total
        ))
        
        return order
    
    @property
    def total(self) -> Decimal:
        """Business logic - calculated property"""
        return sum(item.total for item in self.items)
    
    def pay(self, payment_id: str):
        """Business operation"""
        if self.status != OrderStatus.PENDING:
            raise ValueError("Only pending orders can be paid")
        
        self.status = OrderStatus.PAID
        
        from domain.events import OrderPaid
        self._events.append(OrderPaid(
            order_id=self.id,
            payment_id=payment_id
        ))
    
    def cancel(self, reason: str):
        """Business operation"""
        if self.status not in [OrderStatus.PENDING, OrderStatus.PAID]:
            raise ValueError("Cannot cancel shipped order")
        
        self.status = OrderStatus.CANCELLED
        
        from domain.events import OrderCancelled
        self._events.append(OrderCancelled(
            order_id=self.id,
            reason=reason
        ))


@dataclass
class OrderItem:
    """Value object - part of Order aggregate"""
    product_id: UUID
    product_name: str  # Snapshot
    quantity: int
    unit_price: Decimal
    
    @property
    def total(self) -> Decimal:
        return self.quantity * self.unit_price

Step 2: Define Ports (Interfaces)

# domain/ports/repositories.py
from abc import ABC, abstractmethod
from uuid import UUID
from typing import Optional, List
from domain.models.order import Order

class OrderRepository(ABC):
    """Port: How domain accesses order data
    
    This is an INTERFACE - no implementation
    Business logic depends on this, not on concrete database
    """
    
    @abstractmethod
    async def save(self, order: Order) -> None:
        """Persist an order"""
        pass
    
    @abstractmethod
    async def get(self, order_id: UUID) -> Optional[Order]:
        """Retrieve an order by ID"""
        pass
    
    @abstractmethod
    async def get_by_customer(self, customer_id: UUID) -> List[Order]:
        """Get all orders for a customer"""
        pass
    
    @abstractmethod
    async def delete(self, order_id: UUID) -> None:
        """Delete an order"""
        pass


# domain/ports/gateways.py
from abc import ABC, abstractmethod
from dataclasses import dataclass
from decimal import Decimal

@dataclass
class PaymentResult:
    """Value object for payment result"""
    success: bool
    payment_id: str
    error_message: Optional[str] = None


class PaymentGateway(ABC):
    """Port: How domain processes payments
    
    Interface - can be Stripe, PayPal, or anything
    Business logic doesn't care which
    """
    
    @abstractmethod
    async def charge(
        self,
        amount: Decimal,
        currency: str,
        customer_id: str
    ) -> PaymentResult:
        """Process a payment"""
        pass
    
    @abstractmethod
    async def refund(
        self,
        payment_id: str,
        amount: Decimal
    ) -> PaymentResult:
        """Refund a payment"""
        pass


class EmailService(ABC):
    """Port: How domain sends emails"""
    
    @abstractmethod
    async def send_order_confirmation(
        self,
        email: str,
        order_id: UUID
    ) -> None:
        """Send order confirmation email"""
        pass

Step 3: Implement Business Logic (Service)

# domain/services/order_service.py
from uuid import UUID
from domain.models.order import Order, OrderItem
from domain.ports.repositories import OrderRepository
from domain.ports.gateways import PaymentGateway, EmailService

class OrderService:
    """Domain service - pure business logic
    
    Depends ONLY on ports (interfaces)
    No mention of Django, Stripe, PostgreSQL, etc.
    Can be tested without ANY infrastructure
    """
    
    def __init__(
        self,
        order_repository: OrderRepository,  # Interface, not concrete
        payment_gateway: PaymentGateway,    # Interface, not concrete
        email_service: EmailService,        # Interface, not concrete
    ):
        self.order_repository = order_repository
        self.payment_gateway = payment_gateway
        self.email_service = email_service
    
    async def create_order(
        self,
        customer_id: UUID,
        items: list[OrderItem]
    ) -> Order:
        """Create and pay for an order
        
        Business logic:
        1. Create order (domain)
        2. Save order (via port)
        3. Process payment (via port)
        4. Mark as paid (domain)
        5. Send confirmation (via port)
        """
        
        # 1. Domain logic
        order = Order.create(customer_id=customer_id, items=items)
        
        # 2. Persist via port (don't know if PostgreSQL, MongoDB, etc.)
        await self.order_repository.save(order)
        
        # 3. Process payment via port (don't know if Stripe, PayPal, etc.)
        payment_result = await self.payment_gateway.charge(
            amount=order.total,
            currency="USD",
            customer_id=str(customer_id)
        )
        
        if not payment_result.success:
            raise PaymentFailedException(payment_result.error_message)
        
        # 4. Domain logic
        order.pay(payment_result.payment_id)
        
        # 5. Persist updated state
        await self.order_repository.save(order)
        
        # 6. Send notification via port (don't know if SMTP, SendGrid, etc.)
        await self.email_service.send_order_confirmation(
            email="customer@example.com",  # Would get from customer
            order_id=order.id
        )
        
        return order
    
    async def cancel_order(self, order_id: UUID, reason: str) -> Order:
        """Cancel an order and refund"""
        
        # Get order via port
        order = await self.order_repository.get(order_id)
        
        if not order:
            raise OrderNotFoundException(order_id)
        
        # Domain logic
        order.cancel(reason)
        
        # Persist
        await self.order_repository.save(order)
        
        # Refund via port
        # (Would get payment_id from order)
        await self.payment_gateway.refund(
            payment_id="payment_id",
            amount=order.total
        )
        
        return order


# This service:
# ✓ Can be tested without Django
# ✓ Can be tested without database
# ✓ Can be tested without Stripe
# ✓ Pure business logic
# ✓ Easy to understand
# ✓ Easy to change

# adapters/repositories/django_orm/repositories.py
from domain.models.order import Order, OrderItem, OrderStatus
from domain.ports.repositories import OrderRepository
from .models import OrderModel, OrderItemModel

class DjangoORMOrderRepository(OrderRepository):
    """Adapter: PostgreSQL implementation using Django ORM
    
    Implements OrderRepository port
    Domain doesn't know about this - only knows the interface
    """
    
    async def save(self, order: Order) -> None:
        """Translate domain model to database model"""
        
        # Create or update Django model
        order_model, created = await OrderModel.objects.aupdate_or_create(
            id=order.id,
            defaults={
                'customer_id': order.customer_id,
                'status': order.status.value,
                'created_at': order.created_at,
            }
        )
        
        # Save order items
        # Delete old items, create new ones (simple approach)
        await OrderItemModel.objects.filter(order=order_model).adelete()
        
        for item in order.items:
            await OrderItemModel.objects.acreate(
                order=order_model,
                product_id=item.product_id,
                product_name=item.product_name,
                quantity=item.quantity,
                unit_price=item.unit_price
            )
    
    async def get(self, order_id: UUID) -> Optional[Order]:
        """Translate database model to domain model"""
        
        try:
            order_model = await OrderModel.objects.prefetch_related('items').aget(
                id=order_id
            )
        except OrderModel.DoesNotExist:
            return None
        
        # Reconstitute domain model
        items = [
            OrderItem(
                product_id=item.product_id,
                product_name=item.product_name,
                quantity=item.quantity,
                unit_price=item.unit_price
            )
            async for item in order_model.items.all()
        ]
        
        return Order(
            id=order_model.id,
            customer_id=order_model.customer_id,
            items=items,
            status=OrderStatus(order_model.status),
            created_at=order_model.created_at
        )
    
    async def get_by_customer(self, customer_id: UUID) -> List[Order]:
        """Get all orders for customer"""
        
        orders = []
        async for order_model in OrderModel.objects.filter(
            customer_id=customer_id
        ).prefetch_related('items'):
            order = await self.get(order_model.id)
            if order:
                orders.append(order)
        
        return orders
    
    async def delete(self, order_id: UUID) -> None:
        """Delete order"""
        await OrderModel.objects.filter(id=order_id).adelete()

# adapters/gateways/stripe_gateway.py
import stripe
from decimal import Decimal
from domain.ports.gateways import PaymentGateway, PaymentResult

class StripePaymentGateway(PaymentGateway):
    """Adapter: Stripe implementation
    
    Implements PaymentGateway port
    Can be swapped with PayPalPaymentGateway
    Domain doesn't know which is being used
    """
    
    def __init__(self, api_key: str):
        self.api_key = api_key
        stripe.api_key = api_key
    
    async def charge(
        self,
        amount: Decimal,
        currency: str,
        customer_id: str
    ) -> PaymentResult:
        """Charge customer via Stripe"""
        
        try:
            # Stripe API call
            charge = await stripe.Charge.acreate(
                amount=int(amount * 100),  # Cents
                currency=currency.lower(),
                customer=customer_id
            )
            
            return PaymentResult(
                success=True,
                payment_id=charge.id
            )
        
        except stripe.error.CardError as e:
            # Card declined
            return PaymentResult(
                success=False,
                payment_id="",
                error_message=str(e)
            )
        
        except stripe.error.StripeError as e:
            # Other Stripe error
            return PaymentResult(
                success=False,
                payment_id="",
                error_message=str(e)
            )
    
    async def refund(
        self,
        payment_id: str,
        amount: Decimal
    ) -> PaymentResult:
        """Refund via Stripe"""
        
        try:
            refund = await stripe.Refund.acreate(
                charge=payment_id,
                amount=int(amount * 100)
            )
            
            return PaymentResult(
                success=True,
                payment_id=refund.id
            )
        
        except stripe.error.StripeError as e:
            return PaymentResult(
                success=False,
                payment_id="",
                error_message=str(e)
            )

# config/container.py
from dataclasses import dataclass
from domain.services.order_service import OrderService
from adapters.repositories.django_orm.repositories import DjangoORMOrderRepository
from adapters.gateways.stripe_gateway import StripePaymentGateway
from adapters.gateways.email_service import SMTPEmailService

@dataclass
class Container:
    """Dependency injection container
    
    Wire up concrete implementations here
    This is the ONLY place that knows about concrete classes
    """
    
    # Repositories
    order_repository: OrderRepository
    
    # Gateways
    payment_gateway: PaymentGateway
    email_service: EmailService
    
    # Services
    order_service: OrderService


def create_production_container() -> Container:
    """Production configuration"""
    
    # Create concrete adapters
    order_repo = DjangoORMOrderRepository()
    payment_gateway = StripePaymentGateway(api_key=settings.STRIPE_KEY)
    email_service = SMTPEmailService(host=settings.EMAIL_HOST)
    
    # Inject into service
    order_service = OrderService(
        order_repository=order_repo,
        payment_gateway=payment_gateway,
        email_service=email_service
    )
    
    return Container(
        order_repository=order_repo,
        payment_gateway=payment_gateway,
        email_service=email_service,
        order_service=order_service
    )

# adapters/web/rest/views.py
from django.http import JsonResponse
from django.views import View
from config.container import create_production_container

# Get container (could use dependency injection library)
container = create_production_container()

class CreateOrderView(View):
    """REST API adapter - input adapter
    
    This is infrastructure - translates HTTP to domain
    Business logic is in OrderService, not here
    """
    
    async def post(self, request):
        # 1. Parse HTTP request
        data = json.loads(request.body)
        
        # 2. Convert to domain types
        items = [
            OrderItem(
                product_id=UUID(item['product_id']),
                product_name=item['name'],
                quantity=item['quantity'],
                unit_price=Decimal(item['price'])
            )
            for item in data['items']
        ]
        
        # 3. Call business logic (via service)
        try:
            order = await container.order_service.create_order(
                customer_id=UUID(request.user.id),
                items=items
            )
        except PaymentFailedException as e:
            return JsonResponse(
                {'error': str(e)},
                status=400
            )
        
        # 4. Convert domain to HTTP response
        return JsonResponse({
            'order_id': str(order.id),
            'total': str(order.total),
            'status': order.status.value
        })

Benefits of Hexagonal Architecture

1. Testability

Before (Traditional):

# Cannot test without Django + Database + Stripe
def test_create_order():
    # Needs:
    # - Django test database
    # - Stripe test account
    # - SMTP server
    # Slow (500ms+ per test)
    # Flaky (external dependencies)

After (Hexagonal):

# Test pure logic with mocks
async def test_create_order():
    # No infrastructure needed
    # Fast (<1ms per test)
    # Reliable (no external deps)
    # Focused (business logic only)

2. Flexibility

Before:

# Locked into PostgreSQL
# Changing database = rewrite business logic

After:

# Just implement OrderRepository interface
# Business logic unchanged:
# - PostgreSQLOrderRepository
# - MongoDBOrderRepository
# - RedisOrderRepository
# - InMemoryOrderRepository

3. Independence

Before:

# Business logic depends on:
# - Django ORM
# - Stripe SDK
# - SMTP library
# - HTTP framework
# Tightly coupled

After:

# Business logic depends on:
# - Interfaces only
# Zero infrastructure dependencies
# Can exist in any context

4. Evolution

Before:

# Adding GraphQL = duplicate business logic

After:

# Adding GraphQL = new adapter
# Business logic reused
# REST and GraphQL share same service

Common Patterns

Pattern 1: Repository per Aggregate

# One repository per aggregate root
class OrderRepository(ABC):
    """Order is aggregate root"""
    async def save(self, order: Order): pass
    async def get(self, order_id: UUID): pass

# Order Items accessed through Order
# Not separate repository

Pattern 2: Gateway per External System

# One gateway per external system
class PaymentGateway(ABC): pass
class EmailGateway(ABC): pass
class SmsGateway(ABC): pass
class PushNotificationGateway(ABC): pass

# Clear separation of concerns

Pattern 3: Adapter per Technology

# Multiple adapters for same port
class OrderRepository(ABC): pass

# Implementations:
class PostgreSQLOrderRepository(OrderRepository): pass
class MongoDBOrderRepository(OrderRepository): pass
class InMemoryOrderRepository(OrderRepository): pass

# Choose via configuration

Migration Strategy

Phase 1: Identify Core Domain

Week 1: What is pure business logic?
- Extract domain models
- Identify business rules
- Find bounded contexts

Phase 2: Define Ports

Week 2: What interfaces does domain need?
- Repository interfaces
- Gateway interfaces
- Event bus interface

Phase 3: Implement Adapters

Week 3-4: Create concrete implementations
- Django ORM adapter
- Stripe adapter
- Email adapter
- Keep existing code running

Phase 4: Wire with DI

Week 5: Dependency injection
- Create container
- Configure production adapters
- Configure test adapters

Phase 5: Migrate Incrementally

Week 6+: One feature at a time
- Port one feature to new architecture
- Test thoroughly
- Deploy
- Repeat for next feature

Common Mistakes

Mistake 1: Leaky Abstractions

Problem:

# Port exposing infrastructure detail
class OrderRepository(ABC):
    async def save(self, order: OrderModel):  # ❌ Django model in interface
        pass

Solution:

# Port using domain model
class OrderRepository(ABC):
    async def save(self, order: Order):  # ✓ Domain model
        pass

Mistake 2: Domain Depending on Adapters

Problem:

# domain/services/order_service.py
from adapters.repositories.django_orm import DjangoORMRepository  # ❌

class OrderService:
    def __init__(self):
        self.repo = DjangoORMRepository()  # ❌ Concrete class

Solution:

# domain/services/order_service.py
from domain.ports.repositories import OrderRepository  # ✓

class OrderService:
    def __init__(self, repo: OrderRepository):  # ✓ Interface
        self.repo = repo

Mistake 3: Too Many Ports

Problem:

# Over-engineering
class UserPhotoRepository(ABC): pass
class UserEmailRepository(ABC): pass
class UserNameRepository(ABC): pass
# 50 tiny interfaces

Solution:

# Aggregate-level ports
class UserRepository(ABC):
    # User is aggregate, one repository
    pass

Conclusion

The truth: Dependencies should point inward, not outward.

What we learned:

1. Hexagonal Architecture:

  • Business logic at center
  • Ports define interfaces
  • Adapters implement details
  • Dependencies inverted

2. Benefits:

  • ✓ Testable without infrastructure
  • ✓ Swappable implementations
  • ✓ Framework independent
  • ✓ Technology agnostic
  • ✓ Easy to evolve

3. Implementation:

  • Domain models (pure)
  • Ports (interfaces)
  • Adapters (implementations)
  • Dependency injection (wiring)

4. Testing:

  • Unit tests: Mock ports
  • Integration tests: Real adapters
  • Fast, reliable, focused

Remember:

  • Business logic knows nothing about outside
  • Outside knows about business logic
  • Ports are stable
  • Adapters are replaceable
